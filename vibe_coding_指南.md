# Vibe Coding 指南（重排版易读版）




## 先把话说清楚

氛围编程（Vibe Coding）本身没问题，问题在你。

你听说可以跟 AI 编程助手聊天就能交付软件，就觉得自己成了魔术师。于是你打开 AI，用一句话描述了你的应用想法，然后期待魔法发生。

但出乎意料的是，你得到的是：

- 一堆屎山
- 幻觉出来的 UI 和颜色
- 页面路由不通
- 一个“勉强能用”直到彻底崩掉的 App

然后你怪 AI 犯了错。像个傻子一样。

真相是：AI 产生幻觉，不是因为它坏了，而是因为你没给它任何可以依靠的东西。

你没有给它：

- 结构
- 清晰度
- 基础约束

Vibe Coding 是有效的。

**但前提是：你得理解自己在构建什么，并给 AI 一个真正全面的系统来工作。**

接下来是你要知道的一切：

- 构建模块
- 词汇表
- 真正的工作流程

简单到原始人都能看懂。

如果你读完这篇还是做不出东西，那问题是努力程度，不是信息不足。

这不是一篇扫一眼就忘的推文。这是完整的系统——从头到尾的 Vibe Coding 指南。

现在收藏它，保存到你 Clawdbot 的记忆里。每次开始新项目都回来看。

把这篇当参考手册的人，会构建出惊人的东西。

扫一眼就忘的人，会继续卡壳和崩溃。

现在，让我们来修复你。

---

# 一、为什么你失败了

你失败，是因为你跳过了基础。

你不知道：

- 什么是组件
- 什么是状态（state）
- 为什么按钮点击没反应
- 为什么在你的笔记本上看起来很好，但在手机上就崩了

正因为你不知道这些，所以你无法向 AI 描述它们。

AI 是翻译器。

它把你的意图转换成代码。

但如果你的意图是屎，代码也会是屎。

如果你无法清晰表达想要什么，AI 就会猜。

而猜测会叠加成彻底的混乱。

修复方法不是更好的提示词。

修复方法是：**更好的理解。**

一旦你知道自己在构建什么，提示词就变得微不足道。话语自然流露，因为你终于知道该问什么了。

---

# 二、文档优先系统

这是所有人都做错的地方。

你打开 Cursor，打开聊天，开始描述你的 App，让 AI 立即开始编码。没有计划、没有参考、没有真相来源。

这就是你的项目在前几个文件开始后就崩溃的原因。

真正的系统是：

**文档第一，代码第二。永远这样。**

在写任何一行代码之前，你应该先写项目的规范文档 Markdown 文件。

也就是：

- 清晰
- 具体
- 明确

地描述你在构建什么。

为什么？

因为 AI 编程工具能力高，但确定性低。它们在没有结构护栏的情况下执行任务。

缺乏锁定约束和权威文档，会导致 AI：

- 幻觉需求
- 做出未经授权的架构决策
- 生成解决你从未明确表述过的问题的代码

失败模式不是缺乏编码能力。

失败模式是：**缺乏纪律和上下文保存。**

你应该在碰任何代码之前，写出一套文档栈：

- 六份规范文档，定义整个项目
- 再加上一些让 AI 保持一致、并在会话间保持持久性的文件

---

## 规范文档（你的知识库）

### 1. `PRD.md`（产品需求文档）

完整规格。

你在构建什么、为谁构建、有什么功能、什么在范围内、什么明确在范围外。

应包含：

- 用户故事
- 成功标准
- 非目标
- 每个功能的具体标准

这是你的合同。

AI 读了这个，就知道“完成”对你来说是什么样子。

没有这个？

你不是在构建 App，你是在祈祷一个 App 出现。

### 2. `APP_FLOW.md`

把每个页面和每个用户导航路径，用 plain English 记录下来。

写清楚：

- 什么触发每个流程
- 逐步序列与决策点
- 成功时发生什么
- 错误时发生什么
- 屏幕清单与路由

这能防止 AI 猜测用户如何在 App 中移动。

### 3. `TECH_STACK.md`

把每个包、依赖、API 和工具，锁定到确切版本。

不要歧义。

当 AI 看到“使用 React”，它可能选任何版本。

当它看到：

- `Next.js 14.1.0`
- `React 18.2.0`
- `TypeScript 5.3.3`

它构建的就是你指定的版本。

这份文档消除：

- 幻觉依赖
- 随机技术选择

### 4. `FRONTEND_GUIDELINES.md`

你的完整设计系统。

包含：

- 字体
- 带确切十六进制代码的调色板
- 间距刻度
- 布局规则
- 组件样式
- 响应式断点
- UI 库偏好

每个视觉决策都锁定。

AI 参考这个来创建每个组件。

不再有随机颜色或不一致的间距。

### 5. `BACKEND_STRUCTURE.md`

定义：

- 数据库模式
- 每张表
- 每列
- 类型和关系
- 认证逻辑
- API 端点合约
- 存储规则
- 边缘情况

如果你用 Supabase，这份文档应包含确切的 SQL 结构。

AI 根据这个蓝图构建你的后端，而不是根据它自己的假设。

### 6. `IMPLEMENTATION_PLAN.md`

逐步构建序列。

不是“构建 App”。

而是类似：

- 步骤 1.1 初始化项目
- 步骤 1.2 从 `TECH_STACK.md` 安装依赖
- 步骤 1.3 创建文件夹结构
- 步骤 2.1 按照 `FRONTEND_GUIDELINES.md` 构建导航栏组件

步骤越多，AI 猜测越少。

AI 猜测越少，幻觉越少。

---

## 这些文档如何互相配合

- `PRD` 定义功能
- `APP_FLOW` 定义用户如何体验它们
- `TECH_STACK` 定义用什么构建它们
- `FRONTEND_GUIDELINES` 定义它们长什么样
- `BACKEND_STRUCTURE` 定义数据如何工作
- `IMPLEMENTATION_PLAN` 定义构建顺序

这就是你的知识库。

AI 会读这些并得到它所需的一切。

---

## 两份会话文件（你的持久层）

### `CLAUDE.md`

这是 AI 每次会话自动首先读取的文件。

它包含：

- 每个 AI 会话必须遵循的规则
- 约束
- 模式
- 上下文
- 技术栈摘要
- 文件命名约定
- 组件模式
- 设计系统令牌
- 什么允许、什么禁止

把它当作：

**AI 针对你特定项目的操作手册。**

Claude Code 可以从项目根目录读取这个，甚至不需要你要求。

### `progress.txt`

这是所有人都漏掉的文件。

这个文件跟踪：

- 已完成的内容
- 进行中的内容
- 接下来的内容

每次你完成一个功能，就更新这个文件。

每次你开始新会话、打开新终端窗口、切换分支，AI 首先读取这个文件来获取你进度的上下文记忆。

没有它，每个新会话都从零上下文开始，伴随着一大堆错误。

有了它，AI 精确地从你离开的地方继续。

这就是为什么它很重要：

**AI 在会话间没有记忆。**

当你关闭终端、打开新终端，或开始新聊天时，一切都消失了。

`progress.txt` 是你的外部记忆。

它是会话之间的桥梁。

虔诚地更新它。

在每次实现完成功能后，详细记录：

- 构建了什么
- 什么有效
- 什么坏了
- 接下来做什么

# 三、Interrogation 系统

在你写文档之前，让 AI 把你的想法撕碎。

这是改变一切的提示词：

> 在写任何代码之前，在 Planning 模式下无尽地审问我的想法。不要假设任何问题。问问题直到没有假设剩下。

AI 会在你的清晰度结束时开始产生幻觉。

所以如果你延伸你的清晰度，你就会迫使 AI 在你开始在一个破碎的基础上构建之前，先找出你思维中的 gaps。

---

## AI 应该问你什么

- 这是给谁用的？
- 用户采取的核心行动是什么？
- 他们完成那个行动后发生什么？
- 需要保存什么数据？
- 需要展示什么数据？
- 错误时发生什么？
- 成功时发生什么？
- 这需要登录吗？
- 这需要数据库吗？
- 需要在手机上工作吗？

如果你不能回答这些问题，你还没准备好构建。

先去回答它们。

---

## 一个好的 interrogation 长什么样

假设你在构建一个食谱分享 App：

- **这是给谁用的？** 想要保存和分享食谱的家庭厨师。
- **核心行动？** 用户用标题、食材清单、步骤和照片创建食谱。
- **之后发生什么？** 食谱保存到他们的个人资料并显示在公共 feed 上。
- **保存什么数据？** 食谱标题、食材（数组）、步骤（数组）、照片 URL、作者 ID、时间戳。
- **展示什么数据？** 最近食谱的 feed、单个食谱详情页面、用户自己的食谱集合。
- **错误状态？** 上传失败、缺少必填字段、未授权编辑尝试。
- **需要登录？** 是的，要创建食谱；浏览是公开的。
- **数据库？** 是的，`users` 表和 `recipes` 表带外键关系。
- **手机？** 是的，大多数用户会在做饭时用手机添加食谱。

这些答案不只是答案。

它们是你规范 Markdown 文档的原材料。

- 用户描述喂给你的 `PRD`
- 数据结构喂给你的 `BACKEND_STRUCTURE`
- 流程喂给你的 `APP_FLOW`
- 手机需求喂给你的 `FRONTEND_GUIDELINES`

一旦 interrogation 完成，你对一切都有清晰答案。

然后使用第二个提示词：

> 基于我们的审问，生成我的规范文档文件：`PRD.md`、`APP_FLOW.md`、`TECH_STACK.md`、`FRONTEND_GUIDELINES.md`、`BACKEND_STRUCTURE.md`、`IMPLEMENTATION_PLAN.md`。用我们对话中的答案作为素材。要具体且详尽。没有歧义。

AI 会从审问输出中起草所有文档。

你审查它们，纠正任何模糊的，添加任何缺失的，然后把它们锁定为你的真相来源。

顺序是：

**审问 → 文档 → 代码**

永远不要跳过这些步骤。

# 四、UI vs UX

这是两个每个人都在乱用的术语，但没人简单解释过。

## UI 是什么

UI 是它看起来的样子：

- 颜色
- 字体
- 按钮形状
- 间距

也就是视觉层。

## UX 是什么

UX 是使用起来的感觉：

- 有人能弄清楚要做什么吗？
- 流程直观吗？
- 人们会卡住吗？
- 是沮丧还是愉悦？

## 它们的区别

你可以有漂亮的 UI 和糟糕的 UX。

也可以有丑陋的 UI 和很棒的 UX。

- 漂亮的按钮，但没人知道怎么点
- 功能性的、清晰的界面，人们确切知道要做什么，但它很丑

当你跟 AI 说话时，明确你指的是哪一个。

- “让它看起来更好”是 UI
- “让它更容易用”是 UX
- “让按钮更明显”则两者都有

---

## 一个大多数人不知道的技巧：用截图

用截图作为参考。

找到你喜欢的 App 或网站（可能在 Framer 或 Dribbble 上），截图，直接喂给 Claude Code 或 Cursor。

提示词可以是：

- “匹配这个布局”
- “复现这个确切的 UI”
- “用这个 UI 作为灵感”

AI 可以分析图片并从中提取：

- 设计模式
- 间距
- 调色板
- 排版
- 组件结构

这比用文字描述设计要好无限倍。

Google 的 Gemini 3 Pro，是目前我发现的设计匹配高级美学的最佳模型。

你也可以截图你自己的正在进行的作品，说：

- “这是它现在的样子”
- “这是问题”
- “修复它”

视觉参考比任何书面描述更快地消除歧义。

更好的是：Cursor 有个浏览器侧边栏，让你可以同时设计和编码。

你可以实时看到你的 App 渲染，点击元素、移动东西、更新颜色、测试布局、实时试验 CSS，然后通过 Agent 模式直接把这些更改应用到你的代码库。

这是迭代视觉设计最快的方式，不需要截图、粘贴、等待的循环。

## 你需要知道的设计风格

这些是目前主导的视觉语言。

当你在 `FRONTEND_GUIDELINES.md` 或给 AI 的提示词中引用这些术语时，这些术语会解锁具体、可识别的美学，而不是模糊描述。

### Glassmorphism（玻璃拟态）

磨砂玻璃效果。

特征：

- 带背景模糊的半透明元素
- 微妙的边框
- 柔和阴影漂浮在彩色背景上

想想：

- Apple 的 macOS
- Windows 11
- Spotify 的移动 App

CSS 常见实现：`backdrop-filter: blur()`

适合：

- 卡片
- 模态框
- 导航栏
- 仪表盘

风险：透明度可能降低可读性，所以保持文本对比度高。

### Neobrutalism（新粗野主义）

原始、大胆、故意不精致。

特征：

- 高对比度颜色
- 厚黑边框
- 平阴影
- 冲突的调色板
- 古怪字体

想想：

- Gumroad
- 早期 Notion 的感觉

适合：

- 创意品牌
- 作品集
- 独立工具

提示词可以写：

> 新粗野主义风格，厚边框和粗体原色。

### Neumorphism（软 UI）

元素看起来像是从背景中挤出或压入。

特征：

- 两边柔和的漫射阴影
- 触觉的 3D 感觉

优点：微妙、优雅。

缺点：可访问性棘手，低对比度可能让按钮难以区分。

适合：

- 小 UI 元素
- 切换
- 滑块
- 卡片

不适合作为整个设计语言。

### Bento Grid（便当网格）

模块化布局，内容排列在不同大小的块中，像日本便当盒。

特征：

- 不同尺寸的卡片
- 视觉节奏和层次
- 大卡片放重要内容，小卡片放次要信息

适合：

- 仪表盘
- 产品页面
- 功能展示
- 作品集

这可能是目前最实用的趋势，因为它解决了真正的布局问题。

### Dark Mode（暗黑模式）

不再只是偏好切换，它本身就是一个设计系统。

特征：

- 深色背景配浅色文本
- 仔细设计的对比度比例
- 柔和的强调色

好处：

- 减少眼睛疲劳
- 在 OLED 屏幕上省电
- 看起来高级

如果你在做任何消费级 App，从一开始就规划好亮色和暗黑模式。

不要之后随机添加。

在 `FRONTEND_GUIDELINES.md` 中定义好两个调色板和主题。

### Kinetic Typography（动态排版）

移动、拉伸、响应滚动或光标的文本。

例子：

- 进入时动画的标题
- 滚动时缩放的文本
- 交互式字体处理

不只是淡入。

用现代 CSS 和 Framer Motion 等库，可以在没有繁重 JavaScript 的情况下实现。

适用于：

- Hero 区域
- 关键时刻

### Micro-interactions（微交互）

响应用户操作的小动画。

例子：

- 悬停时微妙缩放的按钮
- 点击时弹跳的复选框
- 感觉“活着”的加载旋转器

这些小细节把抛光产品和真正业余的区分开来。

它们传达：

- 界面是响应的
- 界面是活着的
- 界面是高级的

Framer Motion 和 CSS 过渡就能轻松处理这些。

当你给 AI 提示词时，不要说“让它看起来现代”。

而要说：

> 玻璃拟态卡片 + 便当网格布局 + 暗黑模式 + 悬停状态的微交互。

这是一个具体、可构建的方向。

在编码开始前，把设计决策锁定在 `FRONTEND_GUIDELINES.md` 中：

- 挑 1～2 种设计风格
- 定义调色板
- 定义间距刻度
- 定义圆角
- 定义阴影值
- 定义动画时间

AI 会在它们有扎实文档的情况下，一致地遵循它们。

如果没有文档，每个组件都会看起来不同，零一致性。

# 五、组件

组件是可复用的界面片段。

想象乐高。

每块砖是一个组件。你把它们加在一起构建东西。

例子：

- 按钮是一个组件
- 导航栏是一个组件
- 显示产品的卡片是一个组件
- 表单是由更小组件组成的组件（输入框、标签、按钮）

为什么这对 Vibe Coding 很重要？

当你说“给我构建一个落地页”，AI 必须决定创建什么组件。

如果你不指定，它就会猜。

它可能把所有东西建成一团糟，而不是干净、可复用的片段。

更好的提示词：

> 用这些组件构建落地页：导航栏、hero 区域、功能网格（3 张卡片）、推荐轮播、CTA 区域、页脚。

现在 AI 确切知道要创建什么片段。

每个片段：

- 是隔离的
- 可以独立编辑
- 可以复用

这就是组件化思维的力量。

---

# 六、布局

布局是东西在页面上的位置。

每个网站都是盒子套盒子。

掌握这个概念，你就会理解 90% 的网页设计。

主要盒子：

- 顶部的 header / 导航栏
- 中间的主要内容
- 旁边的可选侧边栏
- 底部的页脚

在主要内容里面，还有更多盒子：

- 分割页面的 sections
- 控制宽度的 containers
- 把物品组织成行和列的 grids
- 分组相关内容的 cards

当你向 AI 描述布局时，可以这样说：

> 两列布局。左边侧边栏，250px 宽。主要内容占据剩余空间。侧边栏固定，不滚动。

这是完整的布局指令。

现在 AI 确切知道要构建什么。

# 七、State（状态）

State 是变化的数据。

当你点击按钮发生事情时，state 变了。
当你在表单打字并看到你的文本出现时，state 变了。
当你切换暗黑模式颜色翻转时，state 变了。

State 是你的 App 感觉活着的原因。

没有 state，一切都是静态的。

常见的 state 例子：

- 菜单是打开还是关闭？
- 用户是登录还是登出？
- 购物车里有什么物品？
- 输入框里有什么文本？
- 这是在加载还是完成了？
- 这是成功还是失败？

当你的按钮没反应时，通常是 state 问题。

点击发生了，但没什么告诉 App 要更新。

当你跟 AI 谈交互性时：

> 当用户点击这个按钮，把 modal state 设为打开。当他们点击 modal 外面，设为关闭。

现在 AI 知道：

- 要跟踪什么 state
- 何时改变它

---

# 八、Styling（样式）

样式是你让东西好看的方法。

CSS 是控制外观的语言：

- 颜色
- 间距
- 字体
- 大小
- 位置

Tailwind 是一个快捷系统。

你不用写 CSS 文件，而是直接给元素加类：

- `bg-blue-500` 让背景变蓝
- `text-xl` 让文本更大
- `p-4` 添加内边距

## Design Tokens（设计令牌）

设计令牌是你能复用的一致值。

例如：

- 你的品牌蓝，不是“某种蓝色”，而是全站统一的十六进制值
- 你的间距不是随机的，而是 4px 的倍数
- 你的圆角在每个卡片上都一样
- 你的阴影值在每个抬升元素上都一样

这就是区分业余 App 和精致 App 的东西。

当每个组件使用相同设计令牌时，你的 App 感觉有凝聚力。

当每个组件发明自己的值时，它看起来像五个人建的。

在开始编码前，把这一切都锁定在 `FRONTEND_GUIDELINES.md` 中：

- 主色、辅色、背景、表面、文本、边框、成功、错误、警告的确切色值
- 间距刻度（4px, 8px, 12px, 16px, 24px, 32px, 48px, 64px）
- 字体栈
- 标题、正文、小文本大小
- 圆角值
- 阴影定义
- 过渡时间

当 AI 有这份文档，它生成的每个组件都会匹配。

没有它，你会花几个小时手动修复那些本不该存在的不一致。

相信我，那是个噩梦。

避免痛苦。

你的样式指令越具体，AI 猜测越少。

- “用 16px 内边距和 8px 圆角把背景做成 `#3B82F6`”
- 永远都比
- “做成蓝色，带一些内边距”

更好。

---

# 九、Responsive Design（响应式设计）

Responsive 意味着：你的网站在所有屏幕尺寸上都工作。

你的笔记本很宽。你的手机很窄。同一个网站，不同的布局。

基本原则：

- 大屏幕上显示更多列
- 小屏幕上垂直堆叠东西
- 大屏幕上显示完整导航
- 小屏幕上显示汉堡菜单

## Breakpoints（断点）

设计变化的屏幕宽度。

例如：

- 手机：0–640px
- 平板：640–1024px
- 桌面：1024px 以上

## Mobile-first（移动优先）

你先为最小屏幕设计，再为大屏幕添加复杂度。

这就是 Tailwind 默认的工作方式：

- 无前缀样式应用于所有屏幕
- 带前缀的样式如 `md:` 和 `lg:` 只在这些断点触发

例如：

`flex flex-col md:flex-row`

意思是：

- 手机上堆叠
- 平板及以上并排

这不是偏好，这是策略。

Mobile-first 会迫使你优先考虑：

- 内容
- 极简主义
- 核心流程

在 `FRONTEND_GUIDELINES.md` 中定义你的断点和响应模式：

- 导航：768px 以下汉堡菜单，以上完整水平导航
- 网格：手机单列、平板两列、桌面三列
- 字体大小：每个断点放大 15%

当 AI 有这些规则文档化，它构建的每个组件都能正确响应各种屏幕尺寸。

总是先考虑手机。

50%+ 的用户可能都在手机上。

如果你只在笔记本上测试，你会为大多数人交付破碎体验。

---

# 十、Pages vs Routes

Page 是用户看到的。
Route 是显示那个 page 的 URL。

例子：

- `yoursite.com/` 显示首页
- `yoursite.com/about` 显示关于页面
- `yoursite.com/products/123` 显示产品 123

Routes 可以是：

- **静态的**：`/about` 总是显示相同页面
- **动态的**：`/products/[id]` 基于 id 显示不同产品

当你向 AI 描述你的 App 时：

> 我需要 4 个页面：首页（`/`）、关于（`/about`）、所有产品（`/products`）、单个产品（`/products/[id]`）。

现在 AI 知道完整结构。

它会：

- 构建正确链接的导航
- 在正确位置创建正确的文件

---

# 十一、Frontend vs Backend

Frontend 是用户看到和交互的东西。

它运行在浏览器里。

Backend 是幕后发生的东西：

- 数据库
- 用户账户
- 处理逻辑
- 服务器端运行的代码

一个表单提交流程通常是：

1. frontend 收集输入
2. 发送到服务器
3. 服务器验证并保存
4. 服务器发回确认
5. frontend 显示成功消息

对于简单网站，你可能不需要 backend。

比如：

- 静态页面
- 没有数据库
- 没有账户

对于有用户、保存数据或复杂逻辑的 App，你需要两者。

要 upfront 告诉 AI 你需要哪个：

- “这只是 frontend，只是一个落地页”
- “这需要带用户账户和数据库的 backend”

这会产生完全不同的输出。

---

# 十二、APIs

API 是两个系统相互交谈的方式。

你的 frontend 需要从 backend 获取数据，它通过 API 去问。

比如：

- 天气 App 通过 API 询问天气服务
- 支付功能通过 API 与 Stripe 通信

把 API 当作服务员。

你告诉服务员你想要什么，服务员去厨房把你的订单带回来。

常见模式：

- `GET` 检索数据
- `POST` 发送数据
- `PUT` 更新数据
- `DELETE` 删除数据

当你跟 AI 交谈时：

> 当页面加载时，向 `/api/products` 发起 GET 请求，并在网格中显示结果。

现在 AI 确切知道要构建什么。

---

# 十三、Databases

数据库是你永久保存东西的地方。

没有数据库，刷新时一切都重置：

- 用户注册，刷新，没了
- 购物车加商品，刷新，没了

对初学者来说，常见选项：

**Supabase 是最简单的开始。**

原因：

- 免费层
- 好文档
- 也处理认证

不确定用什么？

用 Supabase。

## 什么时候需要数据库？

- 用户可以创建账户
- 用户可以保存自己的内容
- 你需要存储提交
- 你有会随时间增长的数据

## 什么时候不需要？

- 静态网站
- 每个人看到相同内容
- 没有用户交互的作品集
- 只通过表单服务收集邮箱的落地页

当跟 AI 交谈时：

> 用 Supabase。我需要一个带 email 和 password 的 `users` 表，以及一个带 `title`、`content` 和 `user_id` 的 `posts` 表。

现在 AI 知道你的数据结构，可以围绕它构建一切。

# 十四、Authentication

Authentication 就是登录 / 登出，也就是证明某人是谁。

这比看起来难得多：

- 密码需要加密
- 会话需要管理
- token 需要处理

很容易搞砸。

**不要从零构建 auth。用服务。**

- `Clerk`：最简单，开箱即有很棒的 UI
- 如果你已经在用 Supabase 做数据库，`Supabase Auth` 也可以

当跟 AI 交谈时：

> 用 Clerk 做认证。用户可以用 email 或 Google 注册。登录后，重定向到仪表盘。

让服务处理困难的部分，你直接接进去。

# 十五、File Types

你会在代码库中看到很多文件。下面是一些常见类型：

- `.html`：网页结构
- `.css`：样式规则
- `.js`：让东西可交互的 JavaScript
- `.jsx`：带 HTML 类似语法的 React JavaScript
- `.ts`：TypeScript，带类型的 JavaScript
- `.tsx`：带 React 的 TypeScript
- `.json`：结构化数据
- `.md`：Markdown，格式化文本
- `.env`：环境变量和密钥
- `.gitignore`：告诉 Git 跳过哪些文件

## 特别重要：`.env`

这是你放 API 密钥、密钥、密码的地方。

规则：

- 永远不要分享这个文件
- 永远不要提交到 git
- 永远不要截图

如果你泄露了 `.env`，你就泄露了对一切的访问。

某个幸运的家伙可能会替你跑出几千美元的 API 账单。

别犯这种错误。

# 十六、Folder Structure

你的文件放在哪里很重要。

混乱的项目会让 AI 困惑。

标准结构如下：

```text
my-app/
├── src/
│   ├── app/            → 页面和路由
│   ├── components/     → 可复用 UI 片段
│   ├── lib/            → 工具、辅助函数
│   └── styles/         → CSS 文件
├── public/             → 图片、静态文件
├── .env                → 密钥（永不分享）
├── CLAUDE.md           → AI 规则和上下文
├── progress.txt        → 会话跟踪
├── PRD.md              → 产品需求
├── APP_FLOW.md         → 用户流程和导航
├── TECH_STACK.md       → 锁定的依赖
├── FRONTEND_GUIDELINES.md → 设计系统
├── BACKEND_STRUCTURE.md   → 数据库和 API 规格
├── IMPLEMENTATION_PLAN.md → 逐步构建顺序
├── package.json        → 依赖
└── README.md           → 项目概览
```

当 AI 生成代码时，告诉它把代码放在哪里。

例如：

> 在 `src/components/Button.tsx` 创建按钮组件。

如果你不指定，AI 可能把文件放在任何地方，最后什么都连不起来。

# 十七、实践中的文档系统

Markdown 不是可选的。

**它是 AI 思考的语言。**

你写的每个 `.md` 文件，都会成为 AI 可以读取、理解和遵循的参考文档。

这就是“文档优先”方法有效的原因。

你不是在为人类写文档。

你是在为 AI 写约束。

## 每份规范文档在构建期间如何使用

### `PRD.md`

范围的事实来源。

当 AI 试图添加你没要求的功能时，你指向 PRD：

> 只构建 `PRD.md` 中的内容。

它能在开始之前就杀死范围蔓延。

### `APP_FLOW.md`

构建页面转换和用户旅程时的参考。

例如：

> 完全按照 `APP_FLOW.md` 第 3 节记录的那样构建登录流程。

AI 就知道：

- 每个屏幕
- 每个重定向
- 每个错误状态

### `TECH_STACK.md`

初始化项目和限制依赖时使用。

例如：

> 只使用 `TECH_STACK.md` 中列出的包。没有询问不要添加新依赖。

这能防止 AI 随机引入你没批准的库。

### `FRONTEND_GUIDELINES.md`

所有前端组件的视觉约束来源。

包括：

- 颜色
- 间距
- 排版
- 组件模式
- 响应规则

例如：

> 按照 `FRONTEND_GUIDELINES.md` 第 2 节样式化这个组件。

### `BACKEND_STRUCTURE.md`

定义你的数据层。

例如：

> 按照 `BACKEND_STRUCTURE.md` 第 2.1 节创建 `users` 表。

AI 构建的会是你指定的模式，而不是它自己的解释。

### `IMPLEMENTATION_PLAN.md`

你的执行序列。

例如：

> 我们在 `IMPLEMENTATION_PLAN.md` 的步骤 4.2。只构建这个步骤。

这能防止 AI 跳步、乱序、乱建。

---

## `CLAUDE.md`：主配置文件

它坐在项目根目录，Claude Code 每次会话都会自动读取它。

它包含所有六份文档的浓缩规则：

- 技术栈摘要
- 命名约定
- 文件结构
- 组件模式
- 禁止的操作

把它当作：

**AI 为自己加载的操作系统手册。**

### 大多数人漏掉的一招

`CLAUDE.md` 应该是一个活文档。

每次 AI 犯错、你纠正它时，都以这句话结束：

> 编辑 `CLAUDE.md`，这样你不会再犯那个错误。

Claude 很擅长为自己写规则。

随着时间推移，你的 `CLAUDE.md` 会变成自我改进的规则手册。

错误率会可测量地下降，因为 AI 在把自己的纠正写进自己的操作规则里。

### 再进一步：`lessons.md`

你还可以加一个 `lessons.md`。

每次：

- 纠正后
- PR 后
- 调试会话后

都让 Claude 用下面两项更新它：

- 导致问题的模式
- 防止它的规则

并让你的 `CLAUDE.md` 指向它：

> 会话开始时审查 `lessons.md` 获取相关项目。

现在 AI 会从它自己在你项目上的历史中学习。

这就是把普通 `CLAUDE.md` 和伟大 `CLAUDE.md` 区分开的自我改进循环。

### 一个 `CLAUDE.md` 示例

- 技术栈：Next.js 14、TypeScript、Tailwind CSS、Supabase
- 所有组件放在 `src/components/`
- 使用带 hooks 的功能组件
- 所有 API 路由放在 `src/app/api/`
- 永远不要使用内联样式，总是用 Tailwind
- 设计令牌：主蓝 `#3B82F6`、背景 `#F9FAFB`
- Mobile-first 响应式方法
- 参考文档：`PRD.md`、`APP_FLOW.md`、`TECH_STACK.md`、`FRONTEND_GUIDELINES.md`、`BACKEND_STRUCTURE.md`、`IMPLEMENTATION_PLAN.md`
- 每次会话开始读取 `progress.txt`
- 完成任何功能后更新 `progress.txt`
- 会话开始时审查 `lessons.md`
- 每次纠正后更新它

当 Claude Code 读取这个时，它在创建的每个文件中都会遵循这些规则：

- 没有漂移
- 没有随机决策
- 只有一致性

### Cursor 的版本

Cursor 有它自己的对应机制：`.cursor/rules`

相同概念，不同工具。

把你的项目规则放在 `.cursor/rules/` 里，Cursor 会在所有模式下自动读取它们。

如果你在同一个项目上同时使用 Cursor 和 Claude Code：

**保持 `CLAUDE.md` 和 Cursor 规则对齐。**

也就是：

- 相同的约束
- 相同的约定
- 相同的真相来源

有些人维护一个，然后复制到另一个。
有些人写一个共享规则文件，然后让两边都引用。

都行。

关键是：

**每个接触你代码库的 AI 工具，都遵循相同规则。**

## `progress.txt`：你的会话桥

在每次功能后更新它。

示例：

```text
已完成：
- 通过 Clerk 的用户认证（登录、注册、Google OAuth）
- 带侧边栏导航的仪表盘布局
- 产品 API 端点（GET /api/products）

进行中：
- 产品详情页面（/products/[id]）
- 需要连接 frontend 到 API

接下来：
- 购物车功能
- Stripe 结账

已知 Bug：
- 点击链接后手机导航不关闭
```

当你：

- 打开新终端
- 开始新 Claude Code 会话
- 切换分支
- 一周后回来

AI 读取这个，就确切知道你在哪里。

没有：

- “我们在做什么？”
- 从零重建上下文
- 第 69 次重新解释整个项目

这就是系统：

- 规范文档定义构建什么
- `CLAUDE.md` 强制执行规则
- `progress.txt` 在会话间保持状态

一起，它们给了 AI 构建项目所需的一切，而不会产生幻觉。

你拥有的 Markdown 文档越多，AI 猜测越少。

AI 猜测越少，你就越强。

# 十八、工具以及如何使用它们

你不需要 50 个工具。

你只需要这些。

但你需要正确使用它们。

不同阶段，用不同工具。

## Cursor

你的代码编辑器。AI 原生，为这种工作流构建。

它看到你的整个项目，理解文件关系，能同时编辑多个文件。

但 Cursor 不只是一个东西，它有四种模式。

### Ask Mode

只读。

AI 读取你的代码库并回答问题，不改变任何东西。

适用场景：

- 探索不熟悉的代码
- 试图理解某事如何工作
- 在碰文件前规划下一步

把它当作你的代码顾问：

- “这个函数做什么？”
- “为什么这个组件重新渲染？”
- “如果我改这个模式会打破什么？”

### Plan Mode

编码前做架构。

你描述你想构建什么，Cursor 创建详细实施计划，提出澄清问题，并能生成视觉图表。

适合在每个新功能开始时使用：

> 我需要给这个 App 加 Stripe 结账。创建一个计划。

你审查计划，调整它，批准步骤，然后把那些步骤发给 Agent Mode 执行。

### Agent Mode

主力模式。

这是 AI 自主：

- 写代码
- 编辑文件
- 运行终端命令
- 安装包
- 修复错误

的地方。

它读取整个项目，遵循你的规则文件，端到端实现功能。

大多数 Vibe Coding，都发生在这里。

### Debug Mode

这是很多人不知道的模式。

当你遇到顽固 bug 时，Debug Mode 不只是随机猜修复。

它会：

- 用运行时日志给代码插桩
- 生成多个假设
- 让你复现 bug
- 测试修复
- 让你验证

这是编辑器里的结构化调试循环。

### Cursor 内部的理想工作流

**Ask 理解 → Plan 架构 → Agent 构建 → Debug 收尾**

无论通过 Claude.ai 还是 Claude Code。

Claude 是你的思考伙伴。

适合做：

- 审问想法

- 处理复杂产品决策
- 起草 `CLAUDE.md`

Claude Code 在终端运行，并自动从项目根目录读取 `CLAUDE.md`。

它会遵循你的规则，不需要你重复。

对于：

- 更大的重构
- 文档繁重的任务
- 多文件架构变更

Claude Code 是极强工具。

## Codex

OpenAI 的编码代理。

这是你的：

- 调试器
- 终结者

Codex 在云端、CLI 或 IDE 中运行。

每个任务会得到自己的隔离沙盒，并预加载你的仓库。

它擅长：

- 阅读代码库
- 跟踪依赖
- 审查配置
- 跨文件提出修复
- 运行测试
- 修复失败
- 迭代直到全部通过

使用时机：

- 架构已成型，但代码在崩
- 发布前需要代码审查
- 想重构而不破坏现有功能
- 需要有人找到你漏掉的 bug

Codex 是终结者。

你也可以并行运行多个 Codex 任务，在不同 bug 或功能上同时推进。

## 十八点五：高级工作流（准备好了再上）

### 用 git worktrees 做并行会话

这是巨大的生产力解锁。

不是一次只做一件事，而是开 3～5 个 git worktrees，每个在自己的 Claude Code 会话里并行运行。

例如：

- 一个 worktree 做认证系统
- 一个做仪表盘布局
- 第三个做 API 端点

它们共享同一个仓库，但独立工作。

过去顺序做一整天的东西，现在可能几小时搞定。

### Plan Mode 作为力量倍增器

让一个 Claude 会话先写计划。

再开第二个会话，让它用高级工程师视角审查计划：

> 审查这个实施计划。找出 gaps、我漏掉的边缘情况、任何会打破的地方。

在写一行代码前，把计划修好。

实施中途出问题时，不要硬推。

切回 plan mode，从当前节点重新规划。

### 用子代理解决复杂问题

当问题大到单个 Claude 会话会在上下文上窒息时，使用子代理。

追加 “use subagents” 到你的请求里。

Claude 会开子会话，分别去研究、探索、并行分析，而主会话保持干净。

这就像你在把工作委托给团队，而不是自己扛全部。

### 自定义技能和斜杠命令

如果你一天会做某件事超过一次，就把它做成可复用技能或 slash command，并提交进 git。

例如：

- `/techdebt`：在每次会话结束时运行，找并清理重复代码
- `context-sync`：拉取最近一周的 Slack、文档、GitHub 活动作为上下文 dump

技能会在多个项目上复利。

### 自主 bug 修复

别总是手把手调 bug。

你拿到 bug 报告后，直接粘给 Claude：

> 修复。

CI 测试失败时：

> 去修复失败的 CI 测试。

把日志、错误跟踪、失败测试给它。

让它自己追根溯源、提出修复。

### 语音听写让提示词更强

你说话比打字快三倍。

而更长、更细致的提示词，往往得到更好的输出。

在 macOS 上，按两次 `fn` 开始听写。

把你真正想要的、平时打字会懒得写的细节说出来。

### 学习模式

如果你想理解 AI 在做什么，而不是只让它做：

开启 Claude Code 的解释性 / 学习型输出风格。

它会解释它做的每个更改背后的推理。

你也可以要求它：

- 生成视觉 HTML 演示来解释陌生代码
- 用 ASCII 画架构图
- 做间隔重复抽认卡

边构建边学习的人，最终会不再需要这篇指南。

# 十九、Git 和版本控制

Git 跟踪你做的每个更改。

没有 git：

- 打破了无法撤销
- 不知道改了什么
- 一个错误可能摧毁一切

有 git：

- 每次更改都被保存
- 你可以回到任何以前版本
- 你的代码住在 GitHub 上，不只在你的电脑里

基础命令：

- `git add .` 暂存更改
- `git commit -m "message"` 提交并描述
- `git push` 上传到 GitHub
- `git pull` 下载最新更改

在 Vibe Coding 时，频繁提交。

例如在每个主要功能完成后：

- “添加了用户登录”
- “添加了产品网格”
- “修复了结账 bug”

如果你打破了什么，总能回去。

这也连接到你的 `progress.txt` 系统：

- 提交代码
- 更新 `progress.txt`
- 一起推送

现在 GitHub 上既有你的代码，也有你的上下文文件。

下一次会话：

- 拉下来
- 读 `progress.txt`
- 继续构建

# 二十、环境变量和密钥

你的 API 密钥、数据库密码和密钥，不应该出现在公共或生产代码库里。

它们应该放在 `.env` 文件里。

然后在代码中通过 `process.env.YOUR_KEY_NAME` 使用。

规则：

- 永远不要提交 `.env` 到 git（把它加进 `.gitignore`）
- 永远不要在聊天里粘贴 API 密钥
- 永远不要截图带密钥的界面
- 永远不要在前端代码中放密钥，只放在后端
- 开发和生产使用不同的密钥

当你部署到 Vercel 时，在它的 dashboard 里手动设置环境变量。

它们不会自动从你的本地 `.env` 转过去。

如果你泄露了密钥：

1. 立刻撤销它
2. 生成新的

# 二十一、部署

你的代码在本地工作了。现在它要在互联网上工作。

Vercel 流程：

1. 推代码到 GitHub
2. 把 GitHub 仓库连接到 Vercel
3. Vercel 自动构建和部署

5. 在 Vercel dashboard 里添加环境变量

> 它在 localhost 上工作，但在 Vercel 上崩了。这是来自 Vercel 日志的错误：……

99% 的部署 bug 都是：

- 缺失 ENV 变量
- 构建设置错误

# 二十二、阅读错误信息

错误不是侮辱。

它们是指令。

错误信息会告诉你：

**到底哪里错了。**

大多数人会恐慌并忽略它们。别这样。

一个错误示例：

```text
TypeError: Cannot read property 'map' of undefined
    at ProductList (src/components/ProductList.tsx:15:23)
```

翻译：

- `TypeError`：你用错了某个东西
- `Cannot read property 'map' of undefined`：你试图在一个不存在的值上调用 `.map()`
- `ProductList.tsx:15:23`：准确指出文件和行号

当你得到错误时，给 AI 一切：

> 我得到这个错误：[完整错误信息]。这是那行的代码：[粘贴相关代码]

你给 AI 的错误上下文越多，修复就越快。

# 二十三、调试循环

当某些东西坏了时：

1. 读错误。真的读。
2. 找位置。什么文件，什么行。
3. 理解声明。错误到底说什么。
4. 检查明显问题：拼写错误、缺失导入、错误变量名。
5. 给 AI 上下文：错误 + 代码 + 你原本期待什么。

循环通常是：

- AI 给你代码
- 你尝试
- 它崩了
- 你粘贴错误
- AI 修复
- 重复直到工作

对于两三轮都修不好的顽固 bug：

- 切到 Cursor 的 Debug mode
- 或启动 Codex

它们尤其擅长跨文件或数据流问题。

记住：

Vibe Coding 从来都不是真正的一次成型。

它是迭代。

真正的技能不是完全避免迭代，而是：

- 快速迭代
- 知道什么时候切工具

# 二十四、发布前验证

发布前，检查这些：

- 它在手机上工作吗？真的在手机上打开它
- 它在不同浏览器中工作吗？
- 没有数据时会发生什么？空状态处理了吗？
- 错误数据时会发生什么？错误状态处理了吗？
- 慢网速时会发生什么？加载状态存在吗？
- 你能不能通过快速点击把它玩坏？
- 密钥在浏览器开发者工具中隐藏了吗？

在你能回答这些问题之前，不要发布。

用户会找到你漏掉的每一个 bug。

一丝不苟地做这个过程。

# 二十六、如何阅读 AI 的输出

AI 给你一些代码。

你知道你在看什么吗？

你不需要理解每一行。

但你需要理解结构：

- 创建了什么文件？
- 它们做什么？
- 它们如何连接？

当 AI 生成代码时，问它：

> 用 plain English 解释你刚构建了什么。每个文件做什么？它们如何连接？

随着时间推移，你会开始识别模式：

- 看到 `import` 就知道它在拉另一个文件
- 看到 `useState` 就知道它在跟踪变化数据
- 看到 API 调用就知道它在取数据

这就是你从“氛围编码者”变成真正构建者的方式。

不是靠记住语法。

而是靠理解模式。

# 二十七、如何迭代

每个人拿到的第一次输出，通常都不是对的。

没关系。

## 迭代系统

1. AI 构建版本 1
2. 你测试，找出哪里错了
3. 你具体描述问题
4. AI 修复
5. 你再测试
6. 重复

### 好的迭代表达

- 产品网格在桌面上显示 4 列，但我需要 3 列
- 卡片图片拉伸了，它们应该是 `object-cover`
- 数据获取时没有加载状态

### 坏的迭代表达

- 它看起来不对，修复它

要具体。永远。

# 二十八、把大想法拆成小碎片

AI 会在巨大而模糊的请求上窒息。

例如：

> 给我构建一个完整的电商网站

这很可能会产出垃圾。

拆成碎片：

1. 设置项目脚手架和安装依赖
2. 构建导航栏组件
3. 构建产品卡片组件
4. 构建产品网格页面
5. 连接数据库并获取产品
6. 构建单个产品页面
7. 加入购物车功能
8. 构建购物车页面
9. Stripe 结账

每个碎片都是：

- 一个任务
- 一段对话
- 一个可独立测试的步骤

这实际上就是你的 `IMPLEMENTATION_PLAN.md` 的作用。

你不是在编码时即时想这些步骤。

你是在开始前就把它编号、排序、写清楚。

然后按步骤执行。

告诉 AI：

> 构建 `IMPLEMENTATION_PLAN.md` 的步骤 5。

而不是：

> 构建下一个东西。

精度会复利。

也别忘了：

每完成一个碎片，就更新 `progress.txt`。

# 二十九、什么时候 AI 是错的工具

有时候你就是得自己学这个东西。

## 用 AI 来做什么

- 生成样板代码
- 写重复逻辑
- 快速探索方法
- 带上下文调试
- 把你的意图翻译成代码

## 你自己要学什么

- 核心概念（也就是这篇指南里的东西）
- 如何阅读 AI 生成的代码
- 如何发现 AI 错了
- 如何在 AI 帮不上时调试
- 你选择的栈在底层是怎么工作的

如果你依赖 AI 做一切，你是在流沙上构建。

一个奇怪 bug 出来，你就会卡住并下沉。

如果 AI 解释错了你还照单全收，那你就完蛋了。

还有一件事：

很多时候，官方网站文档比 AI 更好。

- 文档站给你权威信息
- Stack Overflow 有精确错误的精确答案

AI 擅长综合和生成。

但当它失败时，知道如何查官方资料，是很关键的 fallback 技能。

# 三十、范围和知道何时停止

比糟糕代码更容易杀死项目的，是：

**永无止境的功能列表。**

## 什么叫“完成”

- 核心功能工作
- 用户能完成主要行动
- 常见路径上不崩
- 已部署并可访问

## 什么不叫“还没完成”

- 它还不够“完美”
- 还没处理每一个边缘情况
- 还没做完你想象中的每个功能
- 还没长成 Dribbble 图那样

先发布简单版本，获取反馈。

然后基于真实使用改进。

你从未发布的“最好的 App”，比不上一个已经上线的普通 App。

# 三十一、维护你构建的东西

你构建了它，之后还要改它。

未来的你（或 AI）需要理解代码。

这就是文档系统真正回报的时候。

# 三十三、安全基础

最低要求：

- 永远不要在前端代码中暴露 API 密钥
- 永远在后端验证输入（不要相信浏览器发来的任何东西）
- 使用 HTTPS（Vercel 自动提供）
- 保持依赖更新（旧包会带已知漏洞）
- 用认证服务，而不是自己造认证

你不是在建银行系统。

但你也不该带着明显漏洞把 App 扔上互联网。

至少别给自己挖坑。

# 三十五、Assets、媒体和组件库

你还需要：

- 图片
- 字体
- 图标
- 预构建组件

## 去哪找

- 图标：Lucide React
- 字体：Google Fonts
- 图片：Unsplash（库存图）、你自己的图、AI 生成图

记住：图片文件大小很重要。

大图会拖慢网站。

上传前压缩，尽量使用 Next.js 的 `Image` 组件做自动优化。

## 组件库是大多数初学者不知道的作弊码

你没必要从零写：

- 按钮
- 模态框
- 下拉菜单
- 表单元素

直接用组件库。

### `shadcn/ui`

这是目前 Next.js + Tailwind 生态里最常用的选择之一。

它不是传统依赖包，而是：

**直接把组件源码复制进你的项目。**

这意味着你拥有代码，可以完全自定义。

它提供：

- Button
- Dialog
- Dropdown
- Tabs
- Form
- Data Table

等一系列高质量组件。

底层建立在 Radix UI 原语之上，上层用 Tailwind 样式。

告诉 AI：

> 使用 `shadcn/ui` 组件。用 `npx shadcn@latest init` 初始化，并添加我们需要的组件。

这会节省你大量时间。

在 `TECH_STACK.md` 和 `FRONTEND_GUIDELINES.md` 里明确写明你使用哪个组件库。

这样 AI 才会在整个项目里保持一致。

   - `CLAUDE.md`
   - `progress.txt`
